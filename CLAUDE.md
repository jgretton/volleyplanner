## Project Overview

**VolleyPlanner** (`volleyplanner.co.uk`) is an AI-powered training session planner for volleyball coaches. Coaches describe their session on the homepage, submit the form, and receive a complete structured session plan with court diagrams, drill instructions, and timing — generated by the Claude API.

**Target users**: Volleyball coaches at all levels — club, university, academy, recreational. Time-poor, primarily on mobile during sessions.

**Business model**: Free tier (3 plans/month, account required) + Pro tier (£6/month — unlimited plans, save history, regenerate/swap drills, coach notes, PDF export, share via link).

## Design System

### Design Direction

Elite sports performance aesthetic — dark, minimal, high-impact. Inspired by Nike/Adidas digital experiences, not generic SaaS.
Key principles: dark backgrounds with vivid orange accent used sparingly, bold condensed display type for headings, chips/sliders instead of dropdowns, court line geometry as background texture, no decorative clutter.

### Colour Tokens

Always use CSS custom properties — never hardcode hex values in components.

```css
@theme {
  --color-vp-bg:        #0C1520;   /* near-black — page background */
  --color-vp-surface:   #14202E;   /* card/component surface */
  --color-vp-surface-2: #1C2D40;   /* elevated surface (inputs, nested) */
  --color-vp-border:    #253040;   /* subtle border */
  --color-vp-text:      #E8EEF5;   /* primary text */
  --color-vp-muted:     #7B90A6;   /* secondary/muted text */
  --color-orange:       #FF5820;   /* vivid volleyball orange — accent only */
  --color-navy:         #0C1520;   /* legacy alias = vp-bg */
}
```

**Rule**: `orange` is used only for CTAs, active states, and key highlights. Never as a background fill on large areas.

### Typography

Two fonts — loaded via `next/font/google`, variables set on `<html>`:

- **Display** (`font-display`): Barlow Condensed — weights 600/700/800. Used for all headings. Always paired with `uppercase tracking-tight` and tight line-height (`leading-[0.92]` or `leading-[0.95]`).
- **Body** (`font-sans`): Inter — weights 400/500. Used for all body copy, labels, UI text.

Labels/caps pattern: `text-xs font-medium uppercase tracking-widest text-vp-muted`

### Layout Rules

- Max content width: `max-w-300 mx-auto px-4 sm:px-6 lg:px-8` (`max-w-300` = 1200px in Tailwind v4)
- Mobile-first — every component works at 375px viewport
- Section pattern: `grid-cols-1 lg:grid-cols-[1fr_2fr]` — short bold heading left, content right
- Section dividers: `border-b border-vp-border` — no background colour changes between sections
- **No shadows** — borders only
- Border radius: `rounded-md` for inputs/buttons/chips, `rounded-xl` for cards
- Background texture: subtle SVG court lines (`opacity-[0.05]` to `opacity-[0.08]`) on hero sections only

## TypeScript Rules

- Strict mode always on
- No `any` types — use `unknown` and narrow
- All API route request/response bodies must be typed
- Use Zod for runtime validation of Claude API responses
- Prefer `interface` over `type` for object shapes
- All async functions must handle errors explicitly

---

## API Routes — Conventions

Every API route must:

1. Validate the request body with Zod
2. Return typed responses: `{ data: T }` on success, `{ error: string, code: string }` on failure
3. Use try/catch and never expose stack traces to the client
4. Log errors server-side with `console.error`

Standard error codes:

- `limit_reached` — free tier usage limit hit
- `unauthorized` — not authenticated
- `invalid_input` — validation failed
- `generation_failed` — Claude API error
- `not_found` — resource doesn't exist

---

## Claude API Rules

- Always use model `claude-haiku-4-5-20251001`
- All prompts defined in `lib/claude.ts` — never inline in API routes
- Always validate Claude's JSON response with Zod schema
- If JSON parsing fails, retry once before returning `generation_failed` error
- Never stream — use skeleton UI instead
- Max tokens: 4096 for session plan generation, 1024 for single drill operations

---

## Supabase Rules

- Always use typed client from `lib/supabase/server.ts` in server components and API routes
- Always use typed client from `lib/supabase/client.ts` in client components
- Never use the service role key in client-side code
- All queries must respect RLS — never bypass with service role except in webhooks
- Handle Supabase errors explicitly — check `error` on every query response

## SVG Diagram System

All diagram functions live in `lib/diagrams/`. Each function:

- Takes a `DiagramConfig` object
- Returns a complete SVG string (not a React component)
- Uses `viewBox="0 0 400 500"` — never change these dimensions
- Court: `x=20 y=20 width=360 height=440`, fill `#F0F4F8`, stroke `#94A3B8`
- Net line: `y=240`, stroke `#0B1F3A`, stroke-width 2.5
- Attack lines: `y=147` and `y=333`, stroke `#94A3B8`, stroke-width 1, dashed

**Player marker sizes**: circle `r=14`, label font-size 11px bold, fill white

**Player colours** (hardcoded in diagrams — these don't use CSS vars as SVGs may be exported):

- Server: `#185FA5` (blue)
- Passer/receiver: `#D85A30` (coral)
- Setter: `#16A34A` (green)
- Attacker: `#7F77DD` (purple)
- Defender: `#1D9E75` (teal)
- Coach/cart: `#94A3B8` (grey)

**Arrow styles**:

- Ball trajectory: `stroke="#185FA5"` dashed `stroke-dasharray="6 4"` stroke-width 1.5

## Auth Implementation

### Flow
1. User fills form and hits submit
2. Client checks Supabase session — if not signed in, save intended `/plan?...` path to `sessionStorage` key `vp_redirect_after_auth` and redirect to `/signin`
3. `/signin` page — email input, sends magic link via Supabase, shows generic "check your inbox" confirmation
4. User clicks magic link → `/auth/callback` exchanges code for session, always redirects to `/plan` (no dynamic redirect in callback)
5. `/plan` on mount reads `sessionStorage` for `vp_redirect_after_auth`, validates it, extracts params, clears it, starts generation automatically

### Pages
- `app/signin/page.tsx` — magic link request form, two states: input and confirmation
- `app/auth/callback/route.ts` — already exists, exchanges PKCE code, redirects to `/plan`

### Security Rules — Non-Negotiable

**Open redirect prevention**
- Never redirect to a URL from user input or URL params
- Auth callback always redirects to a hardcoded internal path (`/plan`)
- Before using any stored sessionStorage path, validate: must start with `/`, must not contain `://`
- If validation fails, redirect to `/` silently

**Server-side session validation**
- Never use `supabase.auth.getSession()` in server code — does not revalidate the JWT
- Use `supabase.auth.getClaims()` in server components, API routes, and middleware instead
- Always create a new Supabase client per request on the server

**User enumeration prevention**
- `/signin` always returns the same "check your inbox" message regardless of whether email exists
- Never confirm or deny whether an account exists

**Error handling**
- Log full Supabase auth errors server-side with `console.error`
- Never expose raw Supabase error messages to the client
- Show generic friendly messages only

**Caching**
- All API routes and server components that read auth state must use `cache: 'no-store'`
- Never cache responses that contain session data

**No-account enforcement**
- Users cannot generate plans without being signed in
- Form submit checks session client-side before navigating
- API route also checks `userId` — belt and braces

### Navbar auth state
- Use `createBrowserClient` + `onAuthStateChange` to reactively show sign in / signed in state
- Show "Sign in" when logged out
- Show abbreviated email + "Sign out" button when logged in
- Sign out calls `supabase.auth.signOut()` then redirects to `/`

---

## Error Handling Philosophy

- Never show raw error messages to users
- Always offer a recovery action (retry, go back, contact support)
- Log full errors server-side, show friendly messages client-side
- Network errors: "Something went wrong. Please check your connection and try again."
- Generation errors: "We couldn't generate your plan right now. Please try again."
- Limit errors: "You've used your 3 free plans this month. Upgrade to Pro for unlimited plans."
